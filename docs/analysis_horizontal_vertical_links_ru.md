# Анализ горизонтальных и вертикальных связей в BCC payments service

## 1) Что считается горизонтальными и вертикальными связями

- **Вертикальные связи**: прохождение бизнес-сценария через слои системы сверху вниз (HTTP -> orchestration/service -> domain -> DB/внешние интеграции).
- **Горизонтальные связи**: взаимодействия между компонентами одного уровня (например, service-to-service, route-to-route, внешняя интеграция с несколькими адаптерами, сквозная обработка событий уведомлений).

---

## 2) Вертикальные связи (по основным потокам)

### 2.1 Создание платёжной сессии

**Цепочка слоёв**
1. `POST /payments/create/{deal_id}` в `app/routes/payments.py` вызывает `create_payment_session_for_deal`.  
2. `app/services/payment_service.py` формирует snapshot из Bitrix (`build_payment_snapshot_from_bitrix`), проверяет/переиспользует активную сессию и создаёт новую через DB-слой при необходимости.  
3. DB-слой в `app/db.py` сохраняет сессию в `payment_sessions`, применяет транзакционность и optimistic-lock подход (поле `version` используется при обновлениях).  
4. После фиксации локального состояния сервис синхронизирует ссылку/статус в Bitrix (`sync_session_to_bitrix`).

**Вывод по вертикали**: сценарий чисто проходит сверху вниз, где контроллер тонкий, orchestration сосредоточен в service-слое, а инфраструктурные детали (SQLite и Bitrix HTTP) изолированы в адаптерах.

### 2.2 Переход на оплату (`/pay/{token}`)

**Цепочка слоёв**
1. Route извлекает сессию по токену и вычисляет доменный статус (`session_status`, `is_expired`).  
2. При необходимости формирует BCC payload (`build_bcc_payload_from_session`) и обновляет сессию (`pending`, `bcc_payload_json`, timestamps).  
3. Возвращает HTML-форму редиректа через `render_bcc_redirect_form_with_minfo`.

**Вывод по вертикали**: доменные правила состояния включены до выхода на внешний контур банка; это уменьшает вероятность невалидного редиректа.

### 2.3 Callback/notify от банка

**Цепочка слоёв**
1. `GET|POST /bcc/notify` принимает данные, проверяет Basic Auth, валидирует инварианты уведомления.  
2. Создаётся запись операции с idempotency key в `payment_operations` (защита от дублей).  
3. Выполняется верификация уведомления через банк и расчёт next-state через `apply_payment_callback_update` / `apply_refund_callback_update` (domain-слой).  
4. Итоговое состояние и метаданные фиксируются в `payment_sessions`, операция закрывается как `completed`.

**Вывод по вертикали**: хороший пример event-driven вертикали: ingress webhook -> idempotency barrier -> domain transition -> persistence.

### 2.4 Инициация возврата

**Цепочка слоёв**
1. Endpoint возврата вызывает `mark_refund_for_deal`.  
2. Сумма возврата берётся из Bitrix (`resolve_refund_amount_from_bitrix`), затем domain-валидация допуска возврата (`get_refundable_balance`, текущий `PaymentStatus`).  
3. Создаётся idempotent operation, статус переводится в `refund_pending`.  
4. Отправляется BCC refund/reversal, затем состояние корректируется по результату и/или callback.

**Вывод по вертикали**: возврат сделан двухфазно (локальная фиксация intent + внешняя попытка), что корректно для интеграций с нестабильными внешними API.

## 3) Горизонтальные связи

### 3.1 На уровне маршрутов (API surface)

- В одном router (`app/routes/payments.py`) объединены: пользовательские endpoint’ы (`/pay/...`, `/payments/...`) и банк-специфичные callback endpoint’ы (`/bcc/notify`, `/bcc/backref`).
- Это создаёт **горизонтальную связанность внутри одного транспортного слоя**: бизнес-API и интеграционные webhook-маршруты находятся рядом и разделяют общие helper-функции (логирование, парсинг payload).

### 3.2 На уровне сервисов

- `payment_service` — orchestration-центр; горизонтально связывает:
  - Bitrix sync service,
  - BCC service,
  - DB operations,
  - domain status machine.
- `bcc_service` и `bitrix_sync_service` выступают адаптерами к внешним системам и не содержат основной orchestration (что снижает циклическую зависимость).

### 3.3 На уровне домена и БД

- Domain (`status_machine`) горизонтально используется и routes, и services. Это общий источник правил переходов и расчёта допустимых действий.
- DB-модуль предоставляет единый набор CRUD/transaction функций, которыми пользуются разные сценарии (create/pay/notify/refund), что формирует общую горизонтальную ось консистентности.

### 3.4 Сквозные поперечные связи (cross-cutting)

- Централизованное логирование (`app/core/shared.py`) используется во всех слоях.
- Идемпотентность операций (`payment_operations`) применяется как общий механизм для notify/refund.
- Настройки (`app/settings.py`) — единая конфигурационная зависимость для всех модулей.

---

## 4) Сильные стороны структуры связей

1. **Тонкий transport-слой, толстый service/domain слой** — хорошее разделение ответственности.
2. **Явная state machine** для статусов платежа/возврата повышает предсказуемость поведения.
3. **Идемпотентность и транзакционность** в критичных сценариях интеграции.
4. **Изоляция внешних API** (Bitrix/BCC) в отдельных сервисах-адаптерах.

---

## 5) Риски/узкие места в связях

1. `app/routes/payments.py` очень насыщен логикой и зависимостями; при росте фич может стать «god-router».
2. Сильная роль `payment_service` как orchestration-центра может привести к перегрузке сценариями.
3. Плотная зависимость от структуры полей Bitrix (через settings field codes) — интеграционный риск при изменениях CRM-схемы.
4. SQLite с локальным файлом подходит для текущего масштаба, но вертикаль write-heavy callback’ов может упереться в ограничения concurrency.

---

## 6) Практические рекомендации

1. Разделить routes на несколько роутеров:
   - `payments_public_routes.py` (create/status/pay/refund),
   - `bcc_webhook_routes.py` (notify/backref),
   - `api_compat_routes.py` (legacy/api v1).
2. Вынести notify-flow orchestration из route в отдельный service (`notify_service.py`) для сокращения горизонтальной сложности transport-слоя.
3. Формализовать контракт Bitrix-полей (док + health-check endpoint с проверкой обязательных field codes).
4. Для будущего роста предусмотреть путь миграции SQLite -> PostgreSQL без изменения domain/service API.

---

## 7) Итог

Архитектура уже имеет здоровый каркас: вертикальные потоки читаемы, горизонтальные связи в основном осмысленные и подконтрольные. Главная зона улучшения — уменьшение насыщенности `routes/payments.py` и дальнейшая декомпозиция orchestration по отдельным application-сервисам.
