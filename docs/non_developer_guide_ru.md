# Понятное описание сервиса для не-программистов

Этот документ написан для людей, которые не занимаются разработкой, но участвуют в тестировании, настройке или эксплуатации сервиса.

---

## 1. Что это за сервис

Это сервис, который связывает между собой:
- **Bitrix** — где живёт сделка и данные клиента,
- **BCC** — где происходит оплата или возврат,
- **локальную базу данных** — где сервис хранит техническое состояние платежа.

Проще говоря:
1. из Bitrix берутся нужные данные по сделке,
2. сервис создаёт платёжную ссылку,
3. клиент уходит на страницу банка,
4. банк сообщает результат обратно,
5. сервис обновляет статус у себя и при необходимости записывает его в Bitrix.

---

## 2. Основные термины простыми словами

### Сделка (`deal`)
Карточка сделки в Bitrix, к которой привязан платёж.

### Платёжная сессия (`payment session`)
Техническая запись внутри сервиса: по ней сервис понимает, какой именно платёж был создан, в каком он состоянии и что по нему уже произошло.

### Платёжная ссылка
Ссылка, которую можно отправить клиенту. По ней клиент попадает в банк.

### `ORDER`
Уникальный номер заказа для банка. Он нужен, чтобы банк и сервис понимали, о каком платеже идёт речь.

### `token`
Внутренний идентификатор ссылки внутри сервиса. Он используется в URL вида `/pay/{token}`.

### `notify`
Серверное уведомление от банка о результате операции.

### `backref`
Возврат клиента из банка обратно на сайт/сервис после попытки оплаты. Это не окончательное подтверждение платежа — финальный статус определяется через notify и дополнительную проверку.

### `status check`
Дополнительный запрос сервиса в банк, чтобы убедиться, что notify действительно соответствует реальному состоянию операции.

### `refund`
Возврат денег клиенту.

---

## 3. Какие сценарии умеет сервис

### Создание платежа
Сервис берёт данные из сделки Bitrix и создаёт платёжную ссылку.

### Открытие ссылки клиентом
Когда клиент открывает ссылку, сервис подготавливает форму и отправляет клиента на страницу банка.

### Получение результата от банка
Банк присылает уведомление, а сервис:
- проверяет, что уведомление допустимо,
- при необходимости делает дополнительный запрос в банк,
- сохраняет статус платежа,
- может обновить поля или комментарии в Bitrix.

### Возврат
Сервис может инициировать возврат денег по уже оплаченному платежу.

### Проверка статуса
Сервис умеет вручную запросить состояние платежа в банке.

---

## 4. Какие статусы бывают

### Оплата
- `created` — ссылка создана, но клиент ещё не ушёл в банк.
- `pending` — клиент уже начал процесс оплаты, ожидается подтверждение.
- `paid` — оплата подтверждена.
- `failed` — оплата не подтверждена.
- `expired` — ссылка просрочена.

### Возврат
- `refund_pending` — возврат запущен, ожидается ответ банка.
- `refunded` — возврат подтверждён.
- `refund_failed` — возврат не подтверждён.

Подробная схема переходов описана отдельно в документе `docs/payment_state_machine.md`.

## 5. Какие настройки важны и зачем они нужны

Ниже перечислены не все технические переменные, а именно те, которые важно понимать при настройке.

## 5.1. Общие настройки сервиса

### `APP_NAME`
Имя сервиса. Нужно в основном для логов и служебной информации.

### `PUBLIC_BASE_URL`
Публичный адрес сервиса. По нему формируются ссылки, которые получает клиент.

Пример: если сервис доступен снаружи как `https://pay.example.com`, то это значение должно быть именно таким.

### `DB_PATH`
Путь к локальной базе данных сервиса.

### `PAYMENT_LINK_TTL_MINUTES`
Сколько минут живёт платёжная ссылка.

### `LOG_LEVEL`
Общий уровень логирования приложения.

---

## 5.2. Настройки Bitrix

### `BITRIX_BASE_URL`
Адрес вашего Bitrix.

### `BITRIX_WEBHOOK_TOKEN`
Токен доступа к Bitrix API.

### `BITRIX_USER_ID`
ID пользователя, через которого выполняются вызовы Bitrix webhook API.

---

## 5.3. Какие поля должны быть в Bitrix

Сервис использует специальные поля сделки.

### `BITRIX_FIELD_PAYMENT`
Поле с суммой платежа.

Ожидаемый формат: `сумма|KZT`.

Пример: `1000|KZT`.

### `BITRIX_FIELD_INVOICE`
Номер счёта / документа, который попадёт в описание платежа.

### `BITRIX_FIELD_PRODUCT`
Название продукта или услуги.

### `BITRIX_FIELD_POLICY_TYPE`
Тип полиса или тип продукта, если это используется в вашей логике.

### `BITRIX_FIELD_PAYMENT_URL`
Поле, куда сервис может записать сформированную ссылку на оплату.

### `BITRIX_FIELD_PAYMENT_STATUS`
Поле, куда сервис может записывать статус платежа.

### `BITRIX_FIELD_PAYMENT_ORDER`
Поле, куда сервис может записывать банковский номер заказа (`ORDER`).

### `BITRIX_FIELD_PAYMENT_REFUND_AMOUNT`
Поле с суммой возврата. Его сервис читает, когда запускается возврат.

### `WEBHOOK_SECRET`
Код поля сделки, в котором хранится секретный токен для Bitrix webhook endpoint-ов сервиса.

---

## 5.4. Настройки банка BCC

### `BCC_MERCHANT`
Идентификатор магазина/мерчанта в банке.

### `BCC_MERCH_NAME`
Название магазина, которое использует банк.

### `BCC_TERMINAL`
Идентификатор терминала.

### `BCC_MAC_KEY_HEX`
Секретный ключ банка для подписи запросов.

### `BCC_BACKREF`
Адрес, куда банк вернёт клиента после оплаты.

### `BCC_NOTIFY_URL`
Адрес, куда банк присылает серверное уведомление.

### `BCC_MERCH_URL`
Сайт мерчанта, который передаётся банку.

### `BCC_LANG`
Язык платёжной страницы.

### `BCC_COUNTRY`
Код страны.

### `BCC_BRANDS`
Разрешённые бренды карт.

### `BCC_TRTYPE1_URL`
URL банка для отправки платёжных запросов.

---

## 5.5. Basic Auth для notify

Это защита endpoint-а `/bcc/notify`.

### `BCC_NOTIFY_BASIC_ENABLED`
Главный переключатель.

- `true` — защита включена,
- `false` — защита отключена.

### `BCC_NOTIFY_BASIC_USERNAME`
Логин для входящих уведомлений банка.

### `BCC_NOTIFY_BASIC_PASSWORD`
Пароль для входящих уведомлений банка.

### `BCC_NOTIFY_BASIC_REALM`
Техническое название realm в ответе Basic Auth.

### Когда отключать
Для локальных тестов или временной диагностики, если у вас ещё нет реальных банковских учётных данных.

### Когда включать
Когда сервис подключается к реальному банковскому notify flow.

---

## 5.6. Логи обмена с банком

### `BANK_LOG_FILE`
Файл, куда пишется отдельный лог общения с банком.

По умолчанию это `bcc_bank_exchange.log`.

### `BANK_LOG_FULL_HTTP`
Флаг подробного логирования запросов и ответов банка.

- `false` — более осторожный режим логирования,
- `true` — максимально подробный HTTP-лог для тестов и диагностики.

Если этот флаг включён, в лог можно увидеть:
- куда именно ушёл запрос,
- какой был метод,
- какие были заголовки,
- какое тело запроса отправили,
- какой статус вернул банк,
- какие заголовки вернул банк,
- какой текст или какие поля пришли в ответе.

Это полезно при интеграционных тестах.

### `BANK_LOG_INCLUDE_SENSITIVE`
Флаг, который разрешает писать в bank log чувствительные/персональные данные без маскирования.

- `false` — чувствительные поля маскируются (безопасный режим),
- `true` — данные пишутся как есть (только для короткой диагностики в тестовой среде).

Ограничения безопасности:
- режим срабатывает только если одновременно включён `BANK_LOG_FULL_HTTP=true`,
- режим срабатывает только для тестового мерчанта (`BCC_MERCHANT=00000001`),
- в рабочем контуре этот флаг должен быть выключен.

Если при включённом режиме вы всё ещё видите `***redacted***`, это обычно означает одно из двух:
- не выполнено одно из условий выше,
- часть полей уже приходит замаскированной со стороны банка (например, `CARD_MASK`).

## 6. Какие URL важно знать

### Для пользователей / операторов
- `POST /payments/create/{deal_id}` — создать платёж по сделке.
- `GET /pay/{token}` — открыть ссылку и перейти к оплате.
- `GET /payments/{token}` — посмотреть сохранённый статус сессии.
- `POST /payments/refund/{deal_id}` — запустить возврат.

### Для банка
- `GET|POST /bcc/notify` — серверное уведомление от банка.
- `GET|POST /bcc/backref` — возврат клиента из банка.

### Для Bitrix webhook сценариев
- `GET|POST /api/v1/payments/create`
- `GET|POST /api/v1/payments/status`
- `GET|POST /api/v1/payments/refund`

## 7. Что делать тестировщику

### Если нужно просто проверить создание ссылки
1. Убедиться, что в Bitrix заполнены сумма, счёт, продукт и контакт.
2. Запустить создание платежа.
3. Проверить, что появилась ссылка.
4. Открыть ссылку и убедиться, что происходит переход в банк.

### Если нужно проверить notify
1. Убедиться, что `BCC_NOTIFY_URL` указывает на нужный адрес сервиса.
2. Если это локальный тест без реального банка, при необходимости отключить `BCC_NOTIFY_BASIC_ENABLED`.
3. Если нужен полный разбор интеграции, включить `BANK_LOG_FULL_HTTP=true`.
4. Если нужно видеть данные полностью (без маскирования), временно включить `BANK_LOG_INCLUDE_SENSITIVE=true`.
5. Смотреть лог `bcc_bank_exchange.log`.

### Если нужно проверить возврат
1. Убедиться, что платёж уже подтверждён.
2. Заполнить поле `BITRIX_FIELD_PAYMENT_REFUND_AMOUNT`.
3. Запустить refund endpoint.
4. Проверить новый статус и комментарии в Bitrix.

## 8. Что делать при проблемах

### Проблема: не создаётся ссылка
Проверьте:
- заполнены ли поля сделки в Bitrix,
- корректен ли формат суммы (`amount|KZT`),
- доступен ли Bitrix webhook.

### Проблема: банк не шлёт notify
Проверьте:
- корректен ли `BCC_NOTIFY_URL`,
- доступен ли сервис извне,
- включён ли/не мешает ли Basic Auth,
- есть ли записи в `bcc_bank_exchange.log`.

### Проблема: возврат не проходит
Проверьте:
- был ли платёж реально оплачен,
- заполнена ли сумма возврата,
- что вернул банк,
- есть ли ошибки в bank log.

---

## 9. Если нужна очень короткая памятка

Для тестов обычно важно помнить только это:

- `PUBLIC_BASE_URL` — по какому адресу сервис доступен извне.
- `BCC_NOTIFY_URL` — куда банк присылает уведомления.
- `BCC_NOTIFY_BASIC_ENABLED=false` — можно временно отключить Basic Auth для тестов.
- `BANK_LOG_FULL_HTTP=true` — можно включить максимально подробный лог общения с банком.
- `BANK_LOG_INCLUDE_SENSITIVE=true` — в тестовом мерчанте можно временно логировать чувствительные поля без маскирования.
- `BANK_LOG_FILE` — где смотреть этот лог.
- `BITRIX_FIELD_PAYMENT_URL` / `BITRIX_FIELD_PAYMENT_STATUS` — куда сервис пишет ссылку и статус.
